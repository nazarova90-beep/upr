# Core Beliefs — наши принципы работы

Этот документ — фундамент всего проекта. Здесь зафиксированы убеждения и принципы, которым следуем все: и человек-заказчик, и AI-агенты. Эти принципы не «рекомендации» — это правила.

Принципы вдохновлены статьёй OpenAI ["Harness engineering: leveraging Codex in an agent-first world"](https://openai.com/index/harness-engineering/) (февраль 2026).

## 1. Документация — система записи

**Если решение не записано в `docs/`, его не существует.**

Не полагаемся на:
- чаты с агентом (контекст теряется);
- память человека (всё забывается через неделю);
- Slack/Telegram/Google Docs (агент туда не заглянет);
- комментарии в коде (агент не обязан их найти).

Любое существенное решение → коммит в репозиторий → конкретный файл в `docs/`.

> **Аналогия:** новый сотрудник, пришедший в команду через год, должен суметь понять весь проект, открыв только этот репозиторий. Ничего больше у него нет.

## 2. AGENTS.md — карта, а не энциклопедия

`AGENTS.md` короткий (~100 строк). Это **оглавление**, которое говорит «правила про X — иди туда, правила про Y — иди туда».

Почему не один большой файл со всеми правилами:
- Контекст агента — дефицитный ресурс. Большой файл вытесняет код и саму задачу.
- «Если всё важное — то ничего не важное» — агент перестаёт понимать приоритеты.
- Большой файл устаревает мгновенно и превращается в кладбище.

## 3. Прогрессивное раскрытие (progressive disclosure)

Агент стартует с маленького стабильного входа (`AGENTS.md`) и только при необходимости проваливается глубже по ссылкам. Каждая папка имеет `index.md`-оглавление.

> **Аналогия:** учебник математики начинается со сложения, а не с теоремы Гёделя.

## 4. Boring technology > shiny new

При выборе библиотек и фреймворков отдаём предпочтение **проверенным «скучным»** технологиям. Причины:
- хорошо описаны в данных, на которых обучен AI → агент пишет более правильный код;
- стабильное API → меньше «слома» при обновлениях;
- большое community → проще найти ответы.

Если есть выбор между «проверенным» и «новым модным» — берём проверенное.

## 5. Перед использованием библиотеки — Context7

Перед внедрением **любой** сторонней библиотеки агент **обязан**:

1. Получить актуальную документацию через MCP-сервер `user-context7`.
2. Зафиксировать ключевые правила и примеры использования в `docs/references/<library>.md`.
3. Только потом писать код.

Это правило особенно важно, потому что обучающие данные AI устаревают, а библиотеки обновляются часто.

## 6. Планы — первоклассные артефакты

Каждая нетривиальная задача начинается с **плана** в файле:
- активные планы → `docs/exec-plans/active/`;
- завершённые → `docs/exec-plans/completed/`;
- техдолг → `docs/exec-plans/tech-debt-tracker.md`.

План — это не «формальность», это **рабочий документ**, в который мы дописываем по ходу обсуждения и проставляем галочки.

## 7. Объясняем непрограммисту

Заказчик проекта (Кристина) — **не программист и не имеет технического образования**. Это правило формирует стиль всех документов и общения:

- Технические термины **обязательно** объясняем через простые аналогии (как другу за чаем).
- Шаги делаем маленькими и понятными.
- При показе кода — кратко поясняем, что он делает.
- Используем русский язык как основной для документации продукта.
- Не используем эмодзи без явной просьбы.

## 8. Кодифицируем вкус один раз — применяем автоматически

Если человек заметил «тут сделано неправильно» → правило **превращается в документ или линтер**. В следующий раз агент не должен повторить ошибку.

## 9. Архитектура важнее свободы

Когда дойдём до кода — **строгая слоистая архитектура** с автоматической проверкой границ. Это звучит как бюрократия, но именно это даёт возможность быстро двигаться без хаоса.

> **Аналогия:** жёсткая разметка дороги позволяет ехать быстрее, чем поле без правил.

## 10. Шапка статуса в каждом документе

Каждый `.md` файл начинается с YAML-блока:

```yaml
---
status: draft | in-progress | approved | deprecated
last_updated: YYYY-MM-DD
owner: Кристина
related: ../path/to/related.md
---
```

- **status** — стадия документа.
- **last_updated** — дата последней правки (вручную, чтобы видеть «свежесть»).
- **owner** — кто отвечает за актуальность.
- **related** — ссылки на смежные документы.

Если шапки нет — агент добавляет её при первом редактировании.

## 11. Перед большими изменениями — спрашиваем

Любое **структурное** изменение (переименование папок, перемещение файлов, переписывание ключевого документа) делается **только после явного «да»** от заказчика.

Маленькие правки (добавить пункт, уточнить фразу, обновить дату) — делаем без отдельного разрешения.

## 13. Не добавляем фичи без явного решения

Агент **не имеет права** самостоятельно вводить в продукт функциональность, которой нет в текущей продуктовой стратегии (`docs/product.md`, `docs/product-specs/`).

Если в ходе работы появилась идея новой фичи или изменения поведения:

1. Озвучить идею **в чате** заказчику.
2. Дождаться **явного «да»**.
3. Зафиксировать решение в нужном документе (`docs/product-specs/<feature>.md` или плане в `docs/exec-plans/active/`).
4. Только после этого — реализовывать.

**Аналогия:** строитель не пристраивает к дому веранду «по своей инициативе», даже если она кажется ему хорошей идеей. Сначала — согласование с владельцем дома, потом — стройка.

Это правило особенно важно для AI-агентов, потому что они склонны «достраивать» решение «по аналогии» и могут незаметно расширить скоуп задачи.

## 14. Doc-gardening (на будущее)

Со временем введём отдельный регулярный «обход документации» — агент проверяет:
- какие документы давно не обновлялись;
- где ссылки ведут «в никуда»;
- где статусы устарели.

Пока это делаем вручную при каждом большом обсуждении.

## Эволюция этого документа

Этот файл живой. По мере накопления опыта **добавляем новые принципы** или уточняем существующие. Любое изменение — отдельный коммит с понятным описанием.

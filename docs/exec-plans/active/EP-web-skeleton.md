# Plan: web/ skeleton — Track D close

Closes Phase 1 / Track D of `roadmap.md` § 5: builds the structural skeleton of `web/` (React + Vite + TypeScript) that makes the Phase 1 exit-trigger fully observable — `web/` skeleton runs locally on Mac Safari, Lucent dark theme rendered, route shells in place. Symmetric to `completed/EP-phase1-track-c-skeleton.md` (which delivered `backend/` + frozen `mobile/`). No business logic — file tree first; thin slice (real Gemini call) is Phase 2 of the roadmap, not this plan.

## Goal & Context

`web/` directory does not exist yet. The active client surface for the project moved from `mobile/` (frozen 2026-04-27 by `EP-pivot-to-web.md`) to a new browser app at `web/`. This plan creates `web/` from scratch: the Vite + React + TypeScript scaffold, the routing shell (`/` Workout + `/chat/:exerciseId` Chat), Lucent CSS wired in, i18next initialised with browser language detection, domain TS types ported from `mobile/src/domain/` (which were authored to mirror backend models), and the empty `api/` + `components/` slots placeholders. Mockups (`docs/ui/mockups/*.html`) and the component inventory (`docs/ui/components.md`) are the visual source of truth the route shells render against; no component is implemented in code yet — slots only.

The plan also closes two Lucent gaps inherited from `completed/EP-track-b-mockups.md` § "What was dropped or deferred": (a) author `.bottom-sheet` in `docs/ui/design-system/style.css` and re-render `docs/ui/mockups/technique-popup.html` against it (replaces the current 340 px centered modal fallback); (b) author a token-driven `.icon-btn` family in `style.css` and replace the four inline-styled ad-hoc circular icon buttons (`.exercise-icon-btn`, `.header-icon-btn`, `.input-icon-btn`, `.modal-close`) across all four mockup HTML files. Both gaps are explicitly assigned to `EP-web-skeleton.md` by the predecessor plan; they live in their own phase (Phase 5) so the actual `web/` scaffold (Phases 2–4) is unblocked.

After this plan closes, Phase 1 of `roadmap.md` reaches its full exit-trigger: Track A specs ✅ (2026-04-19), Track B mockups ✅ (2026-04-27), Track C `backend/` skeleton ✅ (2026-04-27), Track D `web/` skeleton ✅ (this plan). Roadmap then advances to Phase 2 — thin slice (separate plan, started after this one is committed).

## Surprises & Discoveries

- **Phase 1 (2026-04-27):** Context7 doesn't index a dedicated `i18next-browser-languagedetector` library — its docs live under the umbrella `/websites/i18next` and `/llmstxt/i18next_llms-full_txt` IDs, which return only the plugin chaining shape and the "do not set `lng` with a detector" rule, not the full default `detection` options object. Cross-checked against the upstream GitHub README + npm registry to recover the complete defaults (`order`, `caches`, `lookupCookie`, etc.). Both sources cited in the new ref's "Source" line. Latest stable confirmed `v8.2.1` (2026-02-12).
- **Plan adjustment (2026-04-27):** `references/index.md` Deprecated table row for `expo-localization.md` still said "Replacement: `i18next-browser-languagedetector` (planned)" — flipped to "Replaced by" to keep the file internally truthful now that the ref is active. One-word fix, same file the phase already mandated to edit.
- **Phase 2 (2026-04-27):** `create-vite@9.0.6` template differs from what the plan expected. (a) `tsconfig.json` is now a thin project-references container (`{"files": [], "references": [{"path": "./tsconfig.app.json"}, {"path": "./tsconfig.node.json"}]}`) — `compilerOptions` live in `tsconfig.app.json` (compiles `src/`) and `tsconfig.node.json` (compiles `vite.config.ts`). The plan's `baseUrl`/`paths` instruction was applied to `tsconfig.app.json` instead. (b) `vite-env.d.ts` is no longer generated; Vite client types come via `"types": ["vite/client"]` in `tsconfig.app.json`. (c) Demo content is richer than the plan listed: extra files `web/src/assets/hero.png`, `web/public/favicon.svg`, `web/public/icons.svg`, plus an updated demo `App.tsx` referencing all of them. All deleted in the strip-demo step; the `<link rel="icon" type="image/svg+xml" href="/favicon.svg" />` line in `index.html` was also removed (would otherwise 404 in Safari console). `<title>` simultaneously bumped from the template default `web` to `UPR` (Phase 3 was going to do this anyway; doing it now avoids leaving the wrong title on disk for one phase). (d) `eslint.config.js` is included by default — no action needed, ESLint stays as Vite ships it (matches Open Question "Whether to install prettier + eslint configs in Phase 2" recommendation: defer prettier).
- **Phase 2 (2026-04-27):** TypeScript 6.0 (which `create-vite@9` pins via `"typescript": "~6.0.2"`) deprecated `compilerOptions.baseUrl` (error TS5101) and now requires relative paths in `paths` when `baseUrl` is omitted (error TS5090: "Non-relative paths are not allowed when 'baseUrl' is not set"). Resolution: drop `baseUrl` entirely and write `"paths": { "@/*": ["./src/*"] }` (note the leading `./`). The Vite alias in `vite.config.ts` is what actually rewrites imports at build time; tsconfig `paths` only feed the editor / `tsc -b` type-check, so the alias still works. Deviation from the plan's literal `"baseUrl": "."` instruction is intentional and documented here.
- **Phase 2 (2026-04-27):** `npm install <pkgs>` invoked from `web/` (via the agent shell `working_directory` parameter) initially wrote `package.json`/`package-lock.json`/`node_modules` to the repo root instead of `web/` — npm walked up the tree because `web/` had no `node_modules` yet and the cwd binding behaved unexpectedly inside the agent sandbox. Cleaned up the stray files at repo root; re-ran inside `web/` using a chained `cd /Users/christinanazarova/Desktop/upr/web && npm install` command, which worked correctly (153 base + 12 runtime packages). Practical lesson for Phase 4 / future phases: when running `npm` against `web/`, always use a `cd web && npm ...` chained shell command, not the `working_directory` argument alone.

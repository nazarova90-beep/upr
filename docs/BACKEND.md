# Backend — серверная часть

Верхнеуровневый обзор серверной части UPR. Полная картина выбранного стека и план масштабирования — в `stack.md`.

## Статус

**Стек выбран:** **Python + FastAPI**, ORM **SQLAlchemy + SQLModel**, БД на MVP **SQLite**. Подробное обоснование, что **не** Django и почему, — в `stack.md` → раздел «Почему именно эти варианты».

## Зона ответственности бэкенда

- Аутентификация и авторизация пользователей.
- Хранение и выдача данных тренировок, упражнений, чатов.
- Приём видео от мобильного клиента, передача в AI, сохранение результатов.
- Биллинг и управление подписками.
- API для мобильного приложения.
- Фоновые задачи (удаление старых сообщений, очистка видео, аналитика).
- **Локализация (после MVP):** учёт предпочитаемого языка пользователя для текстов ошибок, писем, будущих push — см. `FRONTEND.md`, `DATABASE.md`.

## Принципы

1. **Boring tech** — проверенный Python + FastAPI, который хорошо «понимают» AI-агенты.
2. **Слоистая архитектура** с автоматической проверкой границ (см. `../ARCHITECTURE.md`).
3. **Безопасность по умолчанию** (см. `SECURITY.md`).
4. **Observability с первого дня** — структурированные логи; метрики и трейсы подключаются на Этапе 3 (см. `stack.md` → «План расширения»).
5. **Замена внешних звеньев без переписывания** — AI-провайдер и хранилище видео скрыты за абстракциями `ai_provider/` и `storage/`.

## Структура проекта (предварительная, по доменам)

Соответствует доменам из `../ARCHITECTURE.md`:

```
backend/
├── app/
│   ├── main.py
│   ├── core/                # конфиг, логирование, ошибки
│   ├── workout/             # тренировки, упражнения, подходы
│   ├── exercise_chat/       # чаты, сообщения
│   ├── video_analysis/      # приём видео, очередь, пост-обработка
│   ├── ai_coach/            # промпты, диалоговая логика
│   ├── ai_provider/         # абстракция: Gemini / GPT / Claude / локальные
│   ├── storage/             # абстракция: локальная папка / S3
│   └── db/                  # модели SQLAlchemy/SQLModel, миграции
└── tests/
```

## Что входит в MVP-реализацию

- FastAPI-приложение, запускаемое локально.
- Один захардкоженный пользователь (без регистрации).
- Эндпоинт приёма видео + синхронный (через `async`) вызов AI-провайдера.
- Сохранение чатов и метаданных видео в SQLite через ORM.
- Хранение видео — в локальной папке через `storage/`.
- AI-провайдер — реализация поверх **Google Gemini Free Tier**.
- Pose-обработка через **MediaPipe** (на стороне сервера, как пред-процессинг).

Чего на MVP **нет**: регистрация/авторизация, очереди задач, биллинг, push, деплой, админка. Подробнее — в `stack.md` → «Что мы намеренно НЕ берём в MVP».

## План расширения (соответствует этапам в `stack.md`)

- **Этап 2:** простая авторизация (Sign in with Apple/Google), переезд на облачный хостинг, S3 для видео, подключение **SQLAdmin** как минимальной админки.
- **Этап 3:** переход на **PostgreSQL** (через ту же ORM), очередь задач (RQ/Celery), переход на платный AI-тариф, observability (Sentry / OpenTelemetry).
- **Этап 4:** биллинг (App Store IAP / Google Play Billing / Stripe), сервис лимитов.
- **Этап 5:** горизонтальное масштабирование, кэширование, резервный AI-провайдер.

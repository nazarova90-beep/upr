# Reliability — надёжность системы

Этот документ описывает требования к **надёжности и стабильности** UPR: как часто мы можем «лежать», как восстанавливаемся, что мониторим.

## Статус

**Конкретные SLO ещё не зафиксированы.** Появятся к моменту публичного релиза. Пока — общие принципы.

## Принципы

### 1. Сначала работает, потом красиво

Главный фокус MVP — **стабильное выполнение основного сценария** (тренировка → загрузка видео → разбор). Если что-то «ломается» — это блокер релиза.

### 2. Деградация лучше падения

Если AI-разбор недоступен → юзер видит понятное сообщение «временно не работает, попробуйте позже», но **может продолжать тренировку и логировать подходы**. Не блокируем всё приложение из-за одной фичи.

### 3. Видимость → быстрое реагирование

С первого дня — структурированные логи, метрики, алерты. Лучше узнать о проблеме от системы, чем от юзеров.

## Будущие SLO (Service Level Objectives)

Конкретные цифры будут зафиксированы перед публичным релизом. Ориентир:

| Метрика | Целевое значение (черновик) |
|---|---|
| Доступность API | 99.5% (≈3.6 часа простоя в месяц) |
| Среднее время ответа API | < 300 ms |
| Время ожидания разбора видео (p95) | < 60 секунд |
| Успешность загрузки видео | > 99% |
| Успешность разбора видео AI | > 95% |

## Что мониторим

(Будет уточнено при выборе стека.)

- **Логи** — структурированные, с request-id для трассировки.
- **Метрики** — RPS, latency, error rate, очередь анализа видео.
- **Трейсы** — сквозные через все сервисы (в идеале — OpenTelemetry).
- **Алерты** — на критичные метрики (доступность, ошибки, длина очереди).

## Восстановление после сбоев

- **Бэкапы БД** — автоматические, минимум раз в сутки, с возможностью восстановления на конкретный момент времени (point-in-time recovery).
- **Бэкапы видео** — версионирование в объектном хранилище.
- **Disaster recovery план** — отдельный документ перед публичным релизом.
